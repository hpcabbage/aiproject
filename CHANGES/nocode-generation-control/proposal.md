# Proposal

## Background

当前 NoCode 应用生成链路以聊天页单轮流式生成为主，用户发出需求后，前端通过 SSE 接收文本流并在结束后刷新预览。现有链路缺少“本轮生成任务”的显式身份与状态表达，也缺少中途终止、阶段可视化、失败后恢复动作等产品能力。

## Problem

- 用户发起生成后缺少中途终止能力，只能等待流结束。
- 前端只感知 `isGenerating`，无法区分当前是生成中、已终止、失败还是成功。
- 后端缺少“本轮生成”的显式任务状态，不利于 stop / retry / 日志追踪。
- 生成过程是黑盒，用户不知道当前卡在“理解需求 / 生成页面 / 刷新预览”哪一步。
- 失败后缺少面向用户的恢复动作设计。

## Goal

为 NoCode 生成链路补齐一套最小可用的“生成任务控制”能力：

- 每轮生成拥有唯一 `generationId`
- 明确生成任务状态（`generating / stopped / failed / success`）
- 明确阶段（`understanding / generating / refreshing_preview / completed` 等）
- 前端支持终止生成与阶段可视化
- 为后续 retry / 执行详情 / 更强任务控制打基础

## Non-Goals

- 本轮不引入完整多 Agent 编排
- 本轮不强制把“理解需求”拆成独立 Agent
- 本轮不直接建设完整生成记录中心 / 任务历史页
- 本轮不做复杂版本 diff 或自动变更摘要

## Scope

- 后端：补生成任务状态表达、阶段事件、最小 stop 语义设计
- 前端：聊天页增加 stop 按钮、阶段可视化、终止后状态收口
- 协议：SSE 事件从纯文本流升级为支持业务事件
- 文档：补多角色工作流所需 proposal / design / tasks / spec delta

## Acceptance Signals

- 用户在生成中能看到明确阶段，而不是单一 loading
- 用户能手动终止当前生成，界面正确收口
- 新一轮生成不会被上一轮残留流污染
- 失败时前端能区分失败阶段并提供后续动作入口
- 设计层明确哪些能力依赖数据库 / 接口变更

## Risks

- LangChain4j cancel 能力与现有封装之间可能需要额外桥接
- 前端 stop 不等于后端底层请求立即停止，需明确第一版语义
- 若状态设计过重，可能拖慢第一版落地
