# Design - nocode-version-management-v2

## 设计目标

V2 的重点不是把版本管理做成 Git，而是把它做成一个 NoCode 用户看得懂、敢使用、能回溯、能恢复的版本管理中心。

## 核心设计原则

### 1. 产品语义优先
用户看到的应该是：
- 当前使用中
- 稳定版本
- 回滚生成
- 恢复自哪一版

而不是：
- patch
- diff
- 文件变更树

### 2. 快照模型不变
底层继续沿用“真实版本目录 + 数据库索引”模型，不切换为真实 Git。

### 3. 回滚生成新历史
回滚必须生成新版本记录，而不是覆盖旧历史。

### 4. 显式关系优先
来源版本、当前版本、稳定版本等关系，优先使用显式字段表达，而不是前端推断或标题猜测。

## 当前推荐的数据方向

### app 表
- `currentVersionId`（必做）

#### 放在 app 表的原因
`currentVersionId` 表达的是“这个应用当前正在使用哪一个版本”，它属于应用的最新状态，而不是某一条历史版本自己的属性。

把它放在 `app` 表的好处：
- 查询当前版本更直接
- 回滚后只需要更新应用当前指针
- 不需要在版本表里维护“只能有一个 CURRENT”的约束
- 更贴合产品语义：应用当前落在哪一版

### app_frontend_version 表
- `sourceVersionId`（必做）
- `isStable`（必做）
- `versionRole`（当前阶段可不做）

#### 字段说明

##### 1. `sourceVersionId`
含义：当前版本来源于哪一个版本。

推荐规则：
- 手动提交版本：指向提交时应用当前使用的版本
- 回滚生成版本：指向被回滚到的目标版本

作用：
- 前端可展示“恢复自 Vx”
- 可以明确一条版本记录与另一条版本记录的关系
- 比只保留 `parentVersionNo` 更稳，因为它直接指向真实版本 id

##### 2. `isStable`
含义：该版本是否被用户标记为稳定版本。

放在版本表的原因：
- 稳定性是某条历史版本自己的业务属性
- 一个应用可以有多个历史版本，但通常只会对其中少数版本打稳定标记
- 它和当前版本不是同一类概念，不能混到 `app` 表里

##### 3. `versionRole`
当前阶段建议先不落表。

原因：
- `CURRENT` 已经可以由 `app.currentVersionId` 推导
- `ROLLBACK_RESULT` 已经可以由 `sourceType=ROLLBACK` 表达
- `STABLE` 已经可以由 `isStable` 表达
- 现阶段再加 `versionRole` 会造成语义重复

结论：
V2 Phase 2A 先做最小必做字段：
- `app.currentVersionId`
- `app_frontend_version.sourceVersionId`
- `app_frontend_version.isStable`

## 当前推荐 UI 方向

版本管理页升级为“增强版版本中心”：
- 版本号
- 标题/备注
- 创建时间
- 来源标签
- 当前使用中标签
- 稳定版本标签
- 若为回滚结果，展示“恢复自 Vx”

## 分阶段建议

### Phase 2A
先做最小语义层：
- 当前版本
- 稳定版本
- 来源版本

### Phase 2B
再做体验增强：
- 时间线表达
- 更强的版本操作区
