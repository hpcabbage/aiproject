# Release

## 变更摘要
本阶段已把 NoCode 前端版本管理中的“回滚生成新版本”模型，切换为老板要求的 **截断式回滚**。

本次实际落地包括：
1. 重写 change 目标与产品规则，明确回滚后删除后续版本、当前版本直接切回目标版本、后续重新编号；
2. 改造后端 `rollbackToVersion()`，不再创建新的回滚版本记录，而是恢复目标版本、删除后续版本目录与数据库记录，并把 `app.currentVersionId` 直接指回目标版本；
3. 改造前端 `AppVersionPage.vue`，去掉“回滚生成新版本”旧文案，补上危险提示与强确认；
4. 完成真实联调，确认截断式回滚主语义已真实生效。

## 用户可感知效果
用户现在在版本管理页会感知到这些变化：
1. 回滚提示会明确说明：当前版本会直接切到目标版本，目标版本之后的版本会被永久删除；
2. 回滚前会出现强确认，明确提示这是不可恢复的危险操作；
3. 回滚成功后，不会再看到“新生成的回滚版本”，而是直接看到目标版本成为当前版本；
4. 回滚后再次提交时，版本号会从目标版本继续编号，而不是沿旧历史继续累加。

## 已知风险
1. 截断式回滚是破坏性操作，当前最小实现仍存在“文件删除成功但数据库删除失败”或相反的半删除风险；
2. 前端强确认当前使用的是原生 `window.confirm`，风险表达已足够，但交互质感还可以进一步产品化；
3. 真实联调中发现：回滚后新提交版本的 `sourceVersionId` 仍保留为 `1`，主流程不受影响，但该字段语义是否还需要保留，值得后续再判断。

## 提交建议
建议拆成两次提交，保持语义清晰：

1. **后端提交**
   - 标题：`feat：改为截断式版本回滚`

2. **前端提交**
   - 标题：`feat：收口版本页截断式回滚提示`

如果你更想一次收口，也可以合成一个提交：
- `feat：收口前端版本管理的截断式回滚`
