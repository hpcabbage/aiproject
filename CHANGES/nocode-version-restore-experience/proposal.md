# Proposal

## Background
当前 NoCode 已完成前端版本记录第一阶段能力，系统已经具备：
- 用户手动提交前端版本
- 按应用查看版本列表
- 查看版本详情
- 标记稳定版本
- 回滚到指定版本并生成新的回滚版本记录

这意味着“版本记录存在”这件事已经成立，但当前体验仍然更偏向“能力已接通”，还没有完全达到 NoCode 用户能够低风险、低判断成本地使用版本恢复能力的产品化状态。

## Problem
1. 用户在恢复版本前，对目标版本的判断成本仍然偏高；
2. 当前版本、稳定版本、回滚生成版本之间虽然已有标签，但恢复动作前后的信息层级还不够顺；
3. 恢复操作本身的风险提示、确认语义和结果确认还不够强；
4. 对 NoCode 用户而言，版本能力最重要的不只是“有版本”，而是“敢恢复、会恢复、恢复后看得懂当前状态”。

## Goal
在不引入代码 diff 视角的前提下，围绕版本恢复前后的用户体验，补齐 NoCode 版本恢复链路的最小产品化体验，让用户更容易：
- 判断要恢复哪一版；
- 安心执行恢复；
- 在恢复后理解当前版本状态。

## Non-Goals
1. 本轮不做代码级 diff 展示；
2. 本轮不做开发者视图；
3. 本轮不做双版本页面可视化对比；
4. 本轮不扩成完整版本审核流；
5. 本轮不重做底层版本模型与回滚接口。

## Scope
本轮优先覆盖：
- 强化恢复前的版本关键信息展示；
- 强化恢复后的当前状态确认与来源关系展示；
- 在不扩散页面结构的前提下，顺手微调恢复操作的确认语义；
- 保持现有版本模型、接口语义与 NoCode 用户边界不变。

## Acceptance Signals
1. 用户能在恢复前更容易判断目标版本是否正确；
2. 用户能明显感知恢复动作的后果与风险；
3. 恢复完成后，页面能更清楚表达“当前是哪一版、它与目标版本是什么关系”；
4. 整体改动不依赖代码 diff 视图，也不引入新的复杂版本模型。

## Risks
1. 如果范围不收敛，容易扩成完整版本对比产品；
2. 如果恢复前展示过重，可能把 NoCode 用户重新拉回技术细节；
3. 如果恢复后状态表达不够准确，可能让用户误以为旧版本被直接改写而不是生成了新的回滚版本；
4. 如果把恢复中确认语义单独做大，也会冲淡本轮“恢复前判断优先”的主目标。
