# Proposal

## Background
当前 NoCode 已完成前端版本记录第一阶段能力，系统已经具备：
- 用户手动提交前端版本
- 按应用查看版本列表
- 查看版本详情
- 标记稳定版本
- 回滚到指定版本并生成新的回滚版本记录

但老板已重新明确版本恢复的产品语义：当前不需要“回滚生成新版本”的恢复模型，而要切换为更符合普通用户直觉的 **截断式回滚**。

## Problem
当前版本回滚模型与老板最新期望冲突：
1. 当前实现是“回滚后新增一个回滚版本”，版本号持续递增；
2. 老板现在要的是“回滚到某版本后，后续版本全部删除，当前版本直接变成目标版本”；
3. 如果不重定义产品规则、后端逻辑和前端文案，当前版本管理会持续在两套语义之间打架。

## Goal
把当前版本恢复能力改造成 **截断式回滚模型**：
- 回滚到目标版本后，该版本之后的所有版本全部删除；
- 当前版本直接切换为目标版本；
- 后续再次提交时，从目标版本号继续重新编号；
- 前后端对该危险操作提供清晰、强确认的产品表达。

## Non-Goals
1. 本轮不做代码 diff 展示；
2. 本轮不做开发者视图；
3. 本轮不做双版本可视化对比；
4. 本轮不扩成完整版本审核流；
5. 本轮不引入真实 Git 仓库与分支模型。

## Scope
本轮优先覆盖：
- 重写版本回滚产品规则，明确采用截断式回滚；
- 后端回滚逻辑从“生成新版本”切换为“删除后续版本并恢复目标版本”；
- 前端版本页文案、风险提示、回滚弹窗与结果状态同步调整；
- 梳理文件目录与数据库记录在截断式回滚下的删除规则。

## Acceptance Signals
1. 用户回滚到 `V2` 后，`V3+` 不再出现在版本列表中；
2. 当前版本直接切换为 `V2`，而不是新增 `V6` 之类的新回滚版本；
3. 后续再次提交时，版本号从 `V3` 重新开始；
4. 前端回滚确认弹窗能明确提示“将删除哪些版本、不可恢复”；
5. 版本列表、详情区、回滚说明不再出现“回滚生成新版本”这类旧语义。

## Risks
1. 截断式回滚是破坏性操作，误操作风险高；
2. 数据库记录与文件目录删除需要保持一致，否则容易出现半删除状态；
3. 若版本号重用但主键设计不清晰，容易造成版本身份混乱；
4. 若前端仍残留旧文案，会导致用户误判回滚结果。
