# Notes

## T1 版本恢复主路径体验梳理（2026-04-22）

### 当前主路径
基于当前 `AppVersionPage.vue`，用户恢复版本的主路径大致是：
1. 在版本列表中点选一个版本；
2. 在详情区查看版本信息、Meta 信息、版本说明；
3. 在右下侧的“回滚到当前版本”操作卡中填写回滚说明；
4. 点击“回滚到这个版本”；
5. 系统执行回滚，并生成新的回滚版本记录。

### 恢复前 / 恢复中 / 恢复后当前最影响体验的问题

#### 1. 恢复前
最影响体验的问题是：**用户虽然能看到很多字段，但不够快地判断“我现在选中的这一版，到底是不是我想恢复的目标版本”。**

原因：
- 当前详情区信息量不少，但关键信号分散在多个区块；
- “当前版本 / 稳定版本 / 回滚生成 / 来源版本”虽然都有，但没有被收束成一个更直接的恢复决策区；
- 用户在真正点回滚前，还需要自己把标题、标签、来源关系和备注在脑子里重新拼一遍。

#### 2. 恢复中
最影响体验的问题是：**恢复按钮和风险提示已经存在，但“将发生什么”还不够口语化，用户仍需要自己理解“不是改旧版本，而是生成新的回滚版本”。**

原因：
- 当前 warning 文案是对的，但仍偏系统说明；
- “覆盖当前应用输出 + 新生成一条回滚版本记录”这件事没有被足够清楚地强调成一次用户动作结果。

#### 3. 恢复后
最影响体验的问题是：**恢复完成后，用户虽然理论上可以在列表里看到新版本，但页面没有把“你刚刚已经恢复成功，现在当前版本是谁、它和目标版本是什么关系”主动讲清楚。**

原因：
- 当前模型里已经有 `sourceType / sourceVersionId / currentVersion`；
- 但恢复完成后的结果确认仍然偏被动，需要用户自己去重新识别列表状态。

### 当前最值得先做的一个问题
如果只选一个最值得先做的问题，我建议优先选：

**恢复前的目标版本判断成本过高。**

原因：
1. 这是恢复动作发生前的第一道门槛；
2. 判断不准，后面的确认与结果展示都会失去意义；
3. 这是最适合用小改动提升体验的一步，不需要改底层模型或接口；
4. 它也能自然带动恢复中和恢复后表达一起变顺。

### T1 结论
- 恢复前最痛点：用户不够快地判断“当前选中的版本是不是我要恢复的版本”
- 恢复中最痛点：回滚结果语义不够口语化
- 恢复后最痛点：恢复完成后的结果确认不够主动
- 当前最值得先推进的唯一问题：**恢复前目标版本判断成本高**

## T2 第一阶段边界收敛（2026-04-22）

### 本轮只做什么
本轮第一阶段只做两类最小体验增强：
1. **恢复前判断增强**
   - 把当前选中版本的关键信息收束成更容易做恢复决策的展示区；
   - 优先强化标题、状态标签、来源关系、版本备注这几类信息的可读性；
   - 目标是降低用户在点击回滚前的判断成本。
2. **最小结果确认增强**
   - 在恢复完成后，更主动地告诉用户“当前已经切到哪一版，以及这次恢复生成了什么关系的新版本”；
   - 目标是减少用户恢复后再自己扫列表确认状态的负担。

### 本轮明确不做什么
为了防止范围发散，本轮明确不做：
- 代码级 diff 展示
- 开发者视图
- 双版本可视化对比
- 完整版本审核流
- 重做底层回滚接口或版本模型
- 大范围重构整个版本管理页

### 本轮边界判断
当前第一阶段要做的，不是把版本恢复做成完整产品，而是：

**只补“恢复前更容易判断，恢复后更容易确认”这两个最小体验缺口。**

### T2 结论
- 本轮恢复体验第一阶段边界已收敛
- 当前唯一实现重点应落在：
  1. 恢复前目标版本判断
  2. 恢复后最小结果确认
- 恢复中确认语义可以跟着一起微调，但不能扩成独立大改

## T3 最小页面改动方案（2026-04-22）

基于当前 `AppVersionPage.vue`，本轮最小页面改动方案建议只动 3 个区块：

### 区块 1：详情页顶部 `detail-hero`
目标：把“当前选中的版本是不是我要恢复的目标版本”这件事讲得更直接。

建议补法：
- 在现有标题、标签、subtitle 下方新增一个**恢复决策摘要区**；
- 只收束 3 类最关键判断信息：
  1. 当前这是哪一版（版本号 + 标题）
  2. 它和当前版本/稳定版本/来源版本的关系
  3. 恢复它之后会发生什么（会生成新的回滚版本，而不是修改旧版本）

这样用户在进入详情后，不需要先扫完整个 descriptions 才做判断。

### 区块 2：右侧回滚操作卡 `operation-card-warning`
目标：把恢复动作的结果语义讲得更像用户动作，而不只是系统提示。

建议补法：
- 保留现有 warning card 结构；
- 把 warning message 改成更口语化、更结果导向的表达；
- 在按钮附近补一条简短结果提示，例如“恢复后系统会生成一个新的回滚版本，并把它切为当前版本”。

这样做不重构操作区，只增强动作理解。

### 区块 3：恢复后的最小结果确认区
目标：让用户恢复成功后更容易理解“当前已经切到哪一版，以及这次恢复产生了什么关系”。

建议补法：
- 不新增整页成功态；
- 只在当前版本详情区或顶部摘要区预留一块轻量“结果确认提示”文案区域；
- 该区域在识别到当前查看版本属于 `ROLLBACK + currentVersion` 组合时，主动提示：
  - 当前版本是一次恢复生成的新版本
  - 它恢复自哪一版

这样既不需要新增复杂状态机，也能把恢复后最重要的信息主动讲出来。

### 本轮明确不改的区块
- 不重做左侧版本列表结构
- 不重做 `a-descriptions` 为完全卡片化布局
- 不新增双栏对比视图
- 不新增 diff 区块
- 不改接口层和回滚提交逻辑

### T3 结论
本轮最小页面改动方案已收敛为：
1. `detail-hero` 下补恢复决策摘要区
2. 微调回滚操作卡的结果语义
3. 补一个轻量恢复结果确认提示区

这样可以同时覆盖：
- 恢复前判断
- 恢复中理解
- 恢复后确认

但改动面仍然保持在单页局部，不扩成整页重构。
