# Notes

## T1 新需求重定向：从“恢复生成新版本”切换为“截断式回滚”（2026-04-23）

老板已明确最新版本回滚语义：
- 回滚到某个目标版本后，该版本之后的所有版本都应删除；
- 当前版本直接变成目标版本；
- 后续再次提交时，从目标版本号继续重新编号；
- 不再接受“回滚后生成一个新的回滚版本”这套模型。

这意味着当前 change 不能再继续沿用“恢复前判断更轻松、恢复后能理解新回滚版本关系”这条旧方向，而必须改写为：

**围绕截断式回滚，重定义产品规则、后端删除逻辑、前端危险操作确认与状态展示。**

## T2 截断式回滚的产品规则（2026-04-23）

以 `V1 -> V2 -> V3 -> V4 -> V5` 为例：
- 用户回滚到 `V2`
- 回滚后保留：`V1`、`V2`
- 回滚后删除：`V3`、`V4`、`V5`
- 当前版本直接变成 `V2`
- 后续新提交从 `V3` 重新开始

当前结论：
1. 这是“时间线截断式回滚”，不是“保留历史式恢复”；
2. 对用户必须是高风险操作，前端需要强确认弹窗；
3. 后端必须删除数据库记录与版本目录，并把当前输出恢复成目标版本；
4. `id` 继续作为真实唯一标识，`versionNo` 仅作为用户可见编号。

## T3 当前最小实现边界（2026-04-23）

本轮最小实现不应该一口气把所有页面都重构，而应按下面顺序推进：
1. 先改文档与 change 定位，确保多角色工作流对需求理解一致；
2. 再改后端回滚接口语义与执行逻辑；
3. 最后补前端版本页回滚说明、危险弹窗与状态表达。

当前最值得先推进的唯一子步：

**先落一版后端截断式回滚实现设计与任务拆分，避免前端先改文案却挂在旧后端语义上。**
